# Replication Package: MergirafSemi Evaluation

This repository contains the replication package and artifacts for the experiment comparing **MergirafSemi** — a language-agnostic semistructured merge tool — with existing state-of-the-art merge tools.

---

## What This Artifact Provides

This artifact lets users reproduce, end-to-end, the merge-conflict-resolution experiments reported in the paper. It includes:

- The **input datasets** used in the evaluation;
- The **merge tools** evaluated, including MergirafSemi and its variant, packaged as ready-to-run binaries/jars;
- The **scripts and Docker environment** used to clone the subject projects, run the merges, collect raw results, and logs;
- The **analysis scripts** used to compute resolution rates, profile execution phases, and generate the graphs shown in the paper.

Given a Docker installation and a GitHub Personal Access Token, a user with reasonable knowledge of shell scripts and Docker should be able to install and run the experiment without installing any merge tool directly on their host machine. A local Python 3 installation is only needed afterward, to run the analysis scripts that turn the raw execution data into the final tables and graphs.

---

## Repository Structure

The repository is organized to clearly separate the experiment execution environment from the input data, third-party tools, and analysis scripts. The structure is as follows:

- **/analysis/:** Contains all Python scripts used to process, analyze, and visualize the results generated by the experiment (e.g., calculating resolution rates, profiling phases).
  - **/images/:** The destination folder where generated graphs and charts are saved.
- **/executions/:** The output directory. Each time an experiment is run, a new folder is generated here containing the raw results, merged files, and execution logs.
- **/input/:** Contains the datasets and `.csv` files defining the projects and commits used in the evaluation (e.g., `githubGreatestHits`, `topProjects`, `test`). You can customize the evaluation by editing these files.
- **/vendor/:** Contains the executable binaries and dependencies for the third-party merge tools being evaluated: `mergiraf` (baseline), the two MergirafSemi variants — `mergiraf-semi-c`, referred to in the paper as **MergirafSemi**, and `mergiraf-semi-sc`, referred to in the paper as **MergirafSemi+** — and `s3m-all.jar`.
- **Root Files:** Contains the Docker configuration (`Dockerfile`, `entrypoint.sh`) and the main execution shell scripts (`runMergeToolsExperiment.sh`, `runMergeToolsExperimentTest.sh`).

## Requirements

### Software environment

- **Host OS:** Tested on Windows 11 (both the native PowerShell console and WSL2). Linux and macOS are expected to work as well, since the experiment itself runs inside Docker.
- **Docker** (tested with version `28.5.1`, build `e180ab8`; earlier versions are likely compatible, though untested) — the experiment itself (cloning repositories, running the merge tools, collecting logs) runs entirely inside a Docker container. You do **not** need any programming language-related installations, git, or any merge tools installed on your host.
- **Bash** — used to run `runMergeToolsExperiment.sh` / `runMergeToolsExperimentTest.sh`.
- **Git** — required on the host to clone this repository.
- **Python 3.x** — only required on the host if you want to run the scripts in `/analysis/` outside the container.

Inside the Docker image, the following are provisioned automatically (declared in `Dockerfile`):
- **Java 17** (Eclipse Temurin, JDK, Ubuntu Focal base) — required to run `s3m-all.jar`.

No other runtime dependencies are needed beyond Java: the tools under `/vendor/` are shipped as ready-to-run binaries/jars.

### Python dependencies (analysis only)

A `requirements.txt` is provided in `/analysis/` covering the packages actually used by the analysis scripts (`pandas`, `numpy`, `matplotlib`, `seaborn`, `ptitprince`, `requests`, all pinned to confirmed versions — the remaining imports are Python standard library). Install with:

```bash
pip install -r analysis/requirements.txt
```

### Hardware requirements

- **Disk space:** approximately **100 GB** free is recommended for the full evaluation (run via `runMergeToolsExperiment.sh`), since the mining framework runs with the `-k`/`--keep` flag and preserves all cloned repositories under `/executions/<run>/clonedRepositories`.
- **Memory / CPU:** the results reported in the paper were obtained on a dedicated server with an **Intel Xeon E3-1220 @ 3.10GHz (4 cores) and 32 GB of RAM**. The experiment is expected to run on more modest, standard machines as well — execution is single-threaded by default, but this can be changed easily (see the `THREADS` variable in `.env`), so lower-end machines mainly trade off total runtime rather than failing.
- **Network access:** required to clone GitHub repositories and query the GitHub API.

### Credentials

- A **GitHub Personal Access Token (PAT)** is required (set as `GITHUB_ACCESS_KEY` in `.env`). Besides avoiding GitHub API rate limits while cloning repositories, it is also used to push each merge scenario's output back to a repository and trigger a GitHub Actions build/test run, which is a part of the validation process. This build/test validation step is only needed if you want to reproduce the full pipeline described in the paper. For the full pipeline, the token needs permission to push normally to repositories (cloning/pushing branches) **and** to push/trigger GitHub Actions workflows on the destination repository.

## Installation

> **Two ways to use this artifact:** if you just want to inspect, validate, or re-run the **analysis** behind the paper's tables and graphs, you only need step 1 below (clone the repo) plus the Python setup in [Analyzing the Results](#analyzing-the-results) — Docker is not required for this path, since you'll be working directly with the data already provided under `executions/exec-1/`. A GitHub PAT is still needed, though: some analysis scripts pull build data from GitHub Actions (logs, pass/fail status, etc.) to generate CSVs from it. For this analysis-only path, the token only needs **read** access to the destination build repository — the broader push/workflow-trigger permissions described in [Requirements](#requirements) are only needed if you're also running the full pipeline yourself. Reproducing the **full pipeline** (mining scenarios, running merges, and validating each merge output with a real build/test run) requires all the steps below.

1. **Clone this repository:**
   ```bash
   git clone https://github.com/predohenr/experiment-mergirafsemi.git
   cd experiment-mergirafsemi
   ```

2. **Install Docker** and confirm it is available:
   ```bash
   docker --version
   ```

3. **Configure your credentials and settings:** create a local environment file from the provided template and fill in your GitHub PAT.
   ```bash
   cp .env .env.local
   ```
   Then edit `.env.local` and set the following variables:
   - `THREADS` — number of parallel mining threads (default: `1`; increase on multi-core machines to speed up execution).
   - `CONTAINER_NAME_PREFIX` — prepended to the Docker container name, useful for running multiple experiments concurrently without name clashes.
   - `GITHUB_ACCESS_KEY` — your GitHub Personal Access Token.
   - `GITHUB_REPOSITORY` — the destination repository where build/test validation runs are triggered via GitHub Actions; results (pass/fail, logs) are pushed to GitHub and collected afterward, rather than running locally.
   
   Providing the PAT and the GitHub Repository are optional; if you leave them unset, the pipeline still runs, just without the build/test validation step.

4. **Verify the installation** by running the quick validation test. This also builds the Docker image on first run, so it may take a few minutes:
   ```bash
   bash runMergeToolsExperimentTest.sh
   ```
   If everything is configured correctly, a new folder should appear under `/executions/` containing:
   - `clonedRepositories` — the GitHub repositories cloned for the test run (this folder should be empty unless the `--keep`/`-k` flag was passed to the Docker invocation, in which case the cloned repositories are preserved);
   - **mergeAnalysisOutput/** — containing (1) the merge scenario outputs, organized by programming language, repository, commit, file, and tool (in this order), and (2) a `reports` folder with a CSV summary of all scenarios, so manual inspection of individual results isn't required;
   - `output/reports/syntactic-comparison/` — CSVs with the results of the syntactic-normalization comparison step, separated by programming language.

   The presence of these folders, populated as described above, confirms the pipeline is installed and working correctly.

Once the quick test succeeds, you're ready to run the full evaluation.

## Running the Full Experiment

To ensure a highly reproducible environment and avoid conflicts with underlying system dependencies, the entire experiment is containerized using Docker.

Execute the main script to process the complete dataset:
```bash
bash runMergeToolsExperiment.sh
```

Each run creates a timestamped folder under `/executions/` (e.g. `executions/2026_07_16_14_30/`, or a custom name via the `EXPERIMENT_NAME` variable), and mines each supported language (Rust, JavaScript, Go, Python, and Java) in sequence, printing progress such as:
```
Starting Rust Mining...
Starting Javascript Mining...
Starting Go Mining...
Starting Python Mining...
Starting Java Mining...
All experiments finished!
```
The `All experiments finished!` message confirms the full run completed successfully.

> **⚠️ Note on Tool Execution**
> The Docker container acts as a wrapper; it mounts the `/vendor` folder from your host machine into the container, and the results are written back to `/executions/` on the host, so you don't lose data when the container stops.

## Analyzing the Results

Once the experiment is finished, all output data will be available inside the `/executions/` folder. You can use the provided scripts to generate insights and graphs from this raw data.

1. Navigate to the analysis directory:
   ```bash
   cd analysis/
   ```
2. Run the desired Python scripts. For example, to generate runtime performance graphs or evaluate resolution rates:
   ```bash
   python3 runtime_graphs.py
   python3 resolution_rate.py
   ```

*(Ensure you have installed both Python and its dependencies listed in `analysis/requirements.txt` — see [Requirements](#requirements) above.)*

### Reproducing the analysis without re-running the mining

Running the full mining step from scratch requires a GitHub PAT, ~100 GB of disk space, and a non-trivial amount of time. To let reviewers validate the analysis pipeline without re-running the whole experiment, this repository also ships the raw output of the original run under `/executions/exec-1/`, structured as follows:

- **`reports/<lang>/`** (`<lang>` ∈ `js`, `java`, `rs`, `py`, `go`) — one CSV per tool (`s3m`, `mergiraf`, `mergiraf_semi`, `mergiraf_semi_plus`, `diff3`, and `skipped-merge-commits`), where each row records one merge scenario: repository, commit hash, source file path, merge output path, merge status (e.g. `SUCCESS_WITHOUT_CONFLICTS`), and a set of size/timing metrics for that merge.
- **`combined_csv/`** — five `tools_<lang>.csv` files (same five languages), each combining the corresponding per-tool CSVs from `reports/` into a single file. These are generated by running `crossmatch_csv_data.py` inside `reports/`.
- **`combined_csv/output/reports/syntactic-comparison/<lang>/`** — CSVs with the results of the syntactic-normalization comparison step, one set per language.

You can point the analysis scripts directly at this folder to reproduce the paper's tables and graphs.
