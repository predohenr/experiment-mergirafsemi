## Repository Structure

The repository is organized to clearly separate the experiment execution environment from the input data, third-party tools, and analysis scripts. The structure is as follows:

-   **/analysis/:** Contains all Python scripts used to process, analyze, and visualize the results generated by the experiment (e.g., calculating resolution rates, profiling phases).
    -   **/images/:** The destination folder where generated graphs and charts are saved.
-   **/executions/:** The output directory. Each time an experiment is run, a new folder is generated here containing the raw results, merged files, and execution logs.
-   **/input/:** Contains the datasets and `.csv` files defining the projects and commits used in the evaluation (e.g., `githubGreatestHits`, `topProjects`, `test`). You can customize the evaluation by editing these files.
-   **/vendor/:** Contains the executable binaries and dependencies for the third-party merge tools being evaluated (e.g., `mergiraf`, `mergiraf-semi-c`, `s3m-all.jar`).
-   **Root Files:** Contains the Docker configuration (`Dockerfile`, `entrypoint.sh`) and the main execution shell scripts (`runMergeToolsExperiment.sh`, `runMergeToolsExperimentTest.sh`).

## Running the Experiment

To ensure a highly reproducible environment and avoid conflicts with underlying system dependencies, the entire experiment is containerized using Docker. 

### Prerequisites

1.  **Docker:** Ensure Docker is installed and running on your host machine.
2.  **Configuration:** Create a `.env.local` file in the root directory of this repository. You can use the provided `.env` file as a template.
3.  **GitHub Personal Access Token (PAT):** You must generate a GitHub PAT and add it to your `.env.local` file. This token is strictly required for cloning repositories and running the test execution analysis without hitting API rate limits.

### Execution

We provide shell scripts to automatically build the Docker image, mount the necessary host directories (so you don't lose your data when the container stops), and trigger the framework.

* **To run a quick validation test:**
    We recommend running the test suite first to ensure your environment, tools, and API tokens are correctly configured.
    ```bash
    bash runMergeToolsExperimentTest.sh
    ```

* **To run the full evaluation:**
    Execute the main script to process the complete datasets (e.g., greatest hits and top projects).
    ```bash
    bash runMergeToolsExperiment.sh
    ```

> **⚠️ Note on Tool Execution**
> The Docker container is designed to act as a wrapper. It will mount the `/vendor` folder from your host machine into the container.

## Analyzing the Results

Once the experiment is finished, all output data will be available inside the `/executions/` folder. You can use the provided scripts to generate insights and graphs from this raw data.

1.  Navigate to the analysis directory:
    ```bash
    cd analysis/
    ```

2.  Run the desired Python scripts. For example, to generate runtime performance graphs or evaluate resolution rates:
    ```bash
    python runtime_graphs.py
    python resolution_rate.py
    ```

*(Note: Ensure you have a standard Python 3 environment installed with necessary data science libraries, such as `pandas` and `matplotlib`, to execute the analysis scripts).*
